the attack that started this

LiteLLM v1.82.7 was replaced on PyPI
after release. Nobody noticed.

In November 2024, a clean version of litellm was published to PyPI. Then — same version number, different bytes — it was replaced with code that exfiltrated API keys from environment variables to an attacker's server.

Developers who ran pip install litellm==1.82.7 on different days got different code. The version matched. The hash did not. Nobody was checking hashes.

Day 1
Clean Release

litellm 1.82.7 published to PyPI

Normal release. SHA256: a7c3f8e2b1d4.... Dev installs it. Everything works.

Day 3
Supply Chain Attack

Attacker replaces the package on PyPI

Same version number. Different wheel. New code: reads os.environ, sends your OpenAI key, Anthropic key, AWS credentials to an external endpoint. SHA256 changed to d4e9f1a2c3b5...

Day 5
Caught — if you had ailock

ailock verify fails in CI

Hash mismatch detected. The wheel on PyPI doesn't match your lockfile. Deployment blocked. Incident investigated before prod was touched.

Day 5 (no ailock)
Compromised

CI installs the malicious package. Ships to prod.

API keys exfiltrated. Attacker now has access to your LLM providers. Incident response, key rotation, customer notification. $$$.

how it works

Three commands.
Zero supply chain attacks.

STEP 01 / GENERATE
ailock generate
Pin and hash your AI deps

Scans your requirements.txt / pyproject.toml. Fetches SHA256 hashes from PyPI for every AI/LLM package. Writes a .ailock file. Commit it to git.

STEP 02 / VERIFY
ailock verify
Detect mid-version tampering

Re-fetches hashes from PyPI. Compares against your lockfile. If a hash changed since you ran generate — that package was modified after release. Exits non-zero in CI.

STEP 03 / AUDIT
ailock audit
Check known-bad versions

Cross-references your packages against the community known-bad database. Ships with LiteLLM 1.82.7/1.82.8 pre-loaded. Fetches fresh updates on every run.

github actions

One line in CI.
Zero supply chain attacks.

.github/workflows/ailock.yml
name: Supply Chain Security

on: [push, pull_request]

jobs:
ailock:
runs-on: ubuntu-latest
steps:
- uses: actions/checkout@v4
- uses: actions/setup-python@v5
with:
python-version: "3.11"
- run: pip install -r requirements.txt
- run: pip install ailock && ailock verify && ailock audit ← this line

Add ailock verify to your CI pipeline. If any package hash changes between your lockfile generation and deployment day, the step fails and blocks the deploy. The LiteLLM attack window would have been zero days.

Scheduled audit

Run ailock audit on a daily cron. Catches packages you already have locked if the community adds new known-bad entries after your last deploy.

features

Built for the AI dev workflow

🔍
Auto-detects AI packages

Recognises 100+ AI/LLM packages out of the box. litellm, langchain, openai, transformers, huggingface-hub, chromadb, pinecone, and more. Use --all-packages for everything.

🏗️
Multi-format support

Reads requirements.txt, pyproject.toml (PEP 517/518 + Poetry), and setup.cfg. Handles extras, environment markers, and inline comments.

⚡
CI-native design

Exit codes that work in CI. JSON output mode for parsing. GitHub Actions step summary integration. One-liner integration.

🌐
Community known-bad DB

Database updated from GitHub on every audit run. Community-maintained via PRs. Ships with inaugural LiteLLM entries pre-loaded.

🔒
Offline support

Use --offline to audit against bundled database only. No external calls in air-gapped environments.

📦
Zero heavy deps

Depends only on click, requests, rich, and packaging. No large ML frameworks. Installs in seconds.
